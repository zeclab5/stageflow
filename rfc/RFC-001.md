# RFC-001: Database Selection

Version: 0.2
Status: Approved
Date: 2026-07-15
Approved By: Founder
Decision: SQLite (MVP)

---

# 배경

SPEC-310은 Persistence를 추후 결정으로 두고 있다. 현재 repository 인터페이스만 고정되어 있고, 실제 저장소가 없어 이벤트 소싱과 감사 로그 정책을 구현할 수 없다.

---

# 결정

MVP persistence는 SQLite로 진행한다.

## 이유

- 로컬 우선 개발과 빠른 MVP 검증에 적합하다. 별도 서버/운영 인프라 없이 즉시 시작할 수 있다.
- 공연 제작 현장은 오프라인/로컬 환경이 잦아, 파일 기반 저장소가 초기 사용 시나리오와 맞는다.
- Repository 인터페이스 뒤에 격리되어 있으므로, 확장 시점에 DB 교체가 가능하다 (SPEC-300 데이터 정책).
- 6개 Aggregate의 SQLite Repository 구현이 이미 검증되어 있다.

## 대안

- PostgreSQL: 관계형 모델이 Aggregate 단위 저장과 Soft Delete 규격과 일치하고 JSONB로 가변 metadata를 저장할 수 있으나, MVP 단계에서는 운영 인프라 비용이 과하다. 스케일링이 필요한 시점의 1순위 이관 후보로 유지한다.
- MongoDB: 스키마 유연성은 좋으나, 관계형 규칙과 트랜잭션 경계가 복잡해진다.
- Event Store 전용: 의존성과 운영 비용이 크다.

---

# 조건

- 모든 데이터 접근은 Repository 인터페이스를 통해서만 한다. SQLite 종속 코드가 도메인/애플리케이션 레이어로 누출되지 않아야 한다.
- 스키마 변경은 마이그레이션 스크립트로 관리한다.
- PostgreSQL 이관이 필요해지는 시점(멀티 유저 동시 접근, 서버 배포)에 별도 RFC로 이관 계획을 작성한다.

---

# 계획

- SPEC-400 Database에 SQLite 기반 구현 계약(Connection/Provider/Repository/Migration)을 추가한다.
- 앱은 repository 인터페이스만 보고 DB 교체 가능 구조를 유지한다.
